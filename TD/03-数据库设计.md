# 数据库设计(文件3)

> 深化 [`01-总览.md`](01-总览.md) 第 3 节。与实现 `service-data/migrations/schema.sql`、
> `service-data/internal/store/store.go` 保持一致。

## 1. 设计原则

- **字符集**:全库 `utf8mb4`,引擎 `InnoDB`,时间 `DATETIME`(存 UTC,由应用/MySQL `NOW()` 决定)。
- **无用户表**:Telegram 每条消息自带可信 `tg_user_id`,直接作为 owner 列。无认证、无注册、无配额。
- **词条全局去重**:`words` 按归一化文本唯一;释义按词性多行存 `word_senses`。释义是 **DictionaryAPI.dev 缓存**,不是每用户一份。
- **记忆状态每用户一份**:`user_words` 是「(用户 × 单词)」维度,承载 SM-2 状态。
- **可追溯**:每次复习写 `review_logs`,可还原记忆曲线。

## 2. ER 图

```
                    ┌─────────┐        ┌──────────────┐
                    │  words  │1──────N│ word_senses  │   ← DictionaryAPI 缓存(全局)
                    └────┬────┘        └──────────────┘
                         │1
                         │
                         │N
   (tg_user_id) ────────┤              ┌──────────────┐
        │ N             │ user_words   │1────────────N│  review_logs   ← 复习流水
        └───────────────┴──────────────┘              └──────────────┘

   (tg_user_id) ──N  readings         ← LLM 生成的阅读短文留档
```

- `words (1) ─ (N) word_senses`:一个词多个词性/义项。
- `words (1) ─ (N) user_words`:同一个词可被多个用户收藏。
- `user_words (1) ─ (N) review_logs`:一张卡片多次复习。
- `tg_user_id` 是逻辑外键,**不建物理 FK**(无 users 表)。

## 3. 表详解

### 3.1 `words` — 单词词条(全局去重)
| 列 | 类型 | 说明 |
|---|---|---|
| `id` | BIGINT PK | 自增 |
| `text` | VARCHAR(128) **UNIQUE** | 归一化(小写、trim)后的原文 |
| `phonetic` | VARCHAR(64) NULL | 音标,DictionaryAPI 回填 |
| `audio_url` | VARCHAR(255) NULL | 发音音频链接 |
| `created_at` | DATETIME | |

**索引**:`uk_text(text)` —— 查词入口(`GetWordByText`)走它,唯一保证去重。

### 3.2 `word_senses` — 释义/例句缓存
一个词性/义项一行,写入来源为 DictionaryAPI.dev。
| 列 | 类型 | 说明 |
|---|---|---|
| `id` | BIGINT PK | |
| `word_id` | BIGINT FK→words | |
| `pos` | VARCHAR(32) NULL | 词性 partOfSpeech |
| `meaning_en` | VARCHAR(1024) | 英文释义 definition |
| `meaning_cn` | VARCHAR(512) NULL | **纯英英,保留列位不使用** |
| `examples` | JSON NULL | 例句数组 |
| `synonyms` / `antonyms` | JSON NULL | 同/反义词 |
| `source` | VARCHAR(32) | 默认 `dictionaryapi` |
| `created_at` | DATETIME | |

**索引**:`idx_word(word_id)` —— 按词加载所有义项(`loadSenses`)。
**约束**:`fk_sense_word` 保证义项不悬空。
**上限**:抓取侧(`dict.maxSenses=6`)限制每词义项数,防止卡片/存储膨胀。

### 3.3 `user_words` — 单词本 + SM-2 记忆状态(核心表)
| 列 | 类型 | 说明 |
|---|---|---|
| `id` | BIGINT PK | 也是 `review_logs` 的外键目标 |
| `tg_user_id` | BIGINT | owner(逻辑外键) |
| `word_id` | BIGINT FK→words | |
| `ease_factor` | DECIMAL(4,2) 默认 2.50 | SM-2 EF,下限 1.30 |
| `interval_days` | INT 默认 0 | 当前间隔(天) |
| `repetitions` | INT 默认 0 | 连续答对次数 n |
| `due_at` | DATETIME | 下次到期时间(新词=NOW()) |
| `last_review_at` | DATETIME NULL | |
| `status` | TINYINT 默认 0 | 0=新词 1=学习中 2=已掌握 |
| `created_at` / `updated_at` | DATETIME | |

**索引**
- `uk_user_word(tg_user_id, word_id)` —— 幂等收藏(`ON DUPLICATE KEY`),避免重复。
- `idx_due(tg_user_id, due_at)` —— **今日到期查询**的关键复合索引:`WHERE tg_user_id=? AND due_at<=NOW() ORDER BY due_at`,前缀命中且天然有序。

**状态机**:新词收藏 `status=0`;首次复习后置 `1`;当某次算出的 `interval_days >= 21`(`store.masteredInterval`)置 `2`。

### 3.4 `review_logs` — 复习流水
| 列 | 类型 | 说明 |
|---|---|---|
| `id` | BIGINT PK | |
| `user_word_id` | BIGINT FK→user_words | |
| `quality` | TINYINT | 本次自评 0~5 |
| `prev_interval` / `next_interval` | INT | 复习前后间隔 |
| `prev_ef` / `next_ef` | DECIMAL(4,2) | 复习前后 EF |
| `reviewed_at` | DATETIME | |

**索引**:`idx_uw(user_word_id, reviewed_at)` —— 按卡片回放历史 / 画记忆曲线。
写入与 `user_words` 更新在**同一事务**(见 04 §5)。

### 3.5 `readings` — LLM 生成短文留档
| 列 | 类型 | 说明 |
|---|---|---|
| `id` | BIGINT PK | |
| `tg_user_id` | BIGINT | 为谁生成 |
| `content` | TEXT | Gemini 生成的短文(含 `<b>` 高亮) |
| `target_words` | JSON NULL | 文中嵌入的目标词 |
| `model` | VARCHAR(64) NULL | 生成模型标识 |
| `vec_id` | VARCHAR(64) NULL | 后续接 chroma 用,MVP 留空 |
| `created_at` | DATETIME | |

**索引**:`idx_user(tg_user_id, created_at)` —— 按人倒序回看历史阅读。

## 4. 关键查询模式(对照 store.go)

| 场景 | SQL 要点 | 命中索引 |
|---|---|---|
| 查缓存 | `SELECT ... FROM words WHERE text=?` | `uk_text` |
| 加载义项 | `SELECT ... FROM word_senses WHERE word_id=? ORDER BY id` | `idx_word` |
| 幂等收藏 | `INSERT ... ON DUPLICATE KEY UPDATE id=LAST_INSERT_ID(id)` | `uk_user_word` |
| 今日到期 | `WHERE tg_user_id=? AND due_at<=NOW() ORDER BY due_at LIMIT ?` | `idx_due` |
| 提交复习 | `SELECT ... FOR UPDATE` → `UPDATE user_words` + `INSERT review_logs`(事务) | PK |
| 近期词 | `... JOIN words ... WHERE tg_user_id=? ORDER BY created_at DESC LIMIT ?` | — (小结果集) |
| 统计 | `SUM(due_at<=NOW())` / `SUM(status=2)` + `COUNT(review_logs)` | — |

> **N+1 说明**:`GetDueWords` 先查卡片再逐张 `loadSenses`。个人规模 + `LIMIT` 小,可接受;若将来量大,可改为一次 `IN (...)` 批量加载。

## 5. 数据生命周期

1. **首次查词** → 命中?否则 DictionaryAPI 抓取 → 写 `words` + `word_senses`(事务)。
2. **收藏** → `user_words` 幂等插入,`due_at=NOW()`,`status=0`。
3. **复习** → `user_words` SM-2 更新 + `review_logs` 追加(事务)。
4. **阅读** → `readings` 追加。

无删除路径(MVP)。清理/归档策略后续按需加。

## 6. 迁移与运维

- 建表:`mysql wordbot < service-data/migrations/schema.sql`(幂等 `IF NOT EXISTS`)。
- 后续变更建议引入 `golang-migrate`,以有序 `NNN_*.up.sql` 管理。
- 备份:`user_words` + `review_logs` 是不可再生的用户资产,优先备份;`words`/`word_senses` 可从词典重建。
