# 单词本记忆助手 · 设计总览(文件1)

> 一个带**记忆(间隔复习)系统**的智能单词本,通过 **Telegram Bot** 交互。
> 核心功能:**查询单词** · **复习** · **推荐相关阅读**。

本文件是项目总览,包含四大设计板块:

2. [MySQL 数据库表设计](#3-mysql-数据库表设计)

4. [AI Agent 程序设计](#5-ai-agent-程序设计)

后续每个板块可拆分为独立深化文档(见 [文档拆分计划](#8-文档拆分计划))。

## 0. 关键决策(Locked)

| 维度 | 决策 | 说明 |
|---|---|---|
| 交互层 | Python 直接接入 Telegram Bot | 用 `python-telegram-bot`,不经 Go 中转 |
| 用户识别 | **直接用 Telegram `tg_user_id`,无用户系统/无认证** | TG 每条消息自带可信身份;无 `users` 表,`tg_user_id` 下沉到业务表,天然多用户 |
| 使用规模 | 个人 + 朋友,**不做限流/配额** | 无每日上限字段 |
| LLM 用途 | **仅用于「推荐阅读」生成短文** | 查询/复习均不调 LLM;Gemini `gemini-2.5-flash` |
| 检索(RAG) | chroma —— **MVP 不用,后续可选** | 阅读短文由 LLM 按需生成,无需检索真实语料;chroma 留作后续去重/复用 |
| 数据服务 | **Golang** 微服务,持有 MySQL | 承载业务数据与 **SM-2** 复习调度 |
| 复习算法 | **SM-2**(类 Anki) | 记忆系统核心,详见第 3、4 节 |
| 服务间通信 | Python Agent ←**HTTP/JSON REST**→ Go 数据服务 | 本机无 protoc,REST 零代码生成;端点见第 4 节 |
| 词库来源 | **DictionaryAPI.dev 免费词典 API + MySQL 缓存** | 释义/例句**不走 LLM** |
| 释义语言 | **纯英英**,不做中文翻译 | 单词卡片只给英文释义,`meaning_cn` 不使用 |

## 1. 主要功能设计

### 1.1 定位与用户
- **定位**:个人英语单词记忆助手,主打「查了就能记住」。
- **用户**:通过 Telegram 使用的个人学习者(你 + 朋友),以 Telegram `tg_user_id` 区分,**无需注册/登录**。
- **价值**:把「查单词 → 收藏 → 科学复习 → 在真实语境中巩固」串成一条闭环。

### 1.2 三大功能

#### A. 查询(Query)
用户在对话中直接发送一个单词或短语,Bot 返回:
- 音标(含发音音频链接,若有)
- 词性 + **英文释义**(来自 DictionaryAPI.dev)
- 例句(词典自带的 `example` 字段,可能不是每条都有)
- 同义/反义词(词典自带,可选展示)
- 自动把该词加入用户单词本(状态:待复习)

首次查询调 **DictionaryAPI.dev**(`GET https://api.dictionaryapi.dev/api/v2/entries/en/{word}`,免费、无需 Key),解析后写入 `word_senses` 缓存;再次查询同一词命中缓存,零外部调用。
> 本项目定为**纯英英**:卡片只展示英文释义,不做中文翻译。`word_senses.meaning_cn` 保留列位但不使用。

#### B. 复习(Review)—— 记忆系统核心
- 基于 **SM-2** 间隔重复算法,为每个「用户-单词」维护记忆状态(`ease_factor`、`interval`、`due_at`)。
- Bot 主动推送 / 用户 `/review` 触发,取出**今日到期**的词。
- 每张卡片让用户自评记忆质量 `0~5`(通过 Inline 按钮:忘了 / 模糊 / 记得),据此计算下次复习时间。
- 复习记录写入 `review_logs`,可统计记忆曲线。

#### C. 推荐相关阅读(Recommend Reading)
- 取用户近期学习/复习的若干单词,交给 **Gemini 生成一段自然、含这些目标词的定制短文**,并标注文中出现的目标词。
- 生成结果存入 `readings` 表留档(供回看/后续去重)。
- 目的:在真实语境中复现单词,强化记忆。**这是全项目唯一使用 LLM 的环节。**

### 1.3 Telegram 交互设计

| 命令 / 输入 | 行为 |
|---|---|
| 直接发单词,如 `resilient` | 查询并加入单词本 |
| `/review` | 开始今日复习,逐卡推送 + 自评按钮 |
| `/due` | 查看今日待复习数量 |
| `/read` | 推荐一篇含近期单词的阅读 |
| `/list` | 列出单词本(分页) |
| `/stats` | 学习统计(总词数、今日复习、掌握率) |
| `/help` | 帮助 |
| 自然语言,如「resilient 什么意思」 | Agent 意图识别后走查询链路 |

## 3. MySQL 数据库表设计

> 字符集统一 `utf8mb4`,引擎 `InnoDB`,时间统一 `DATETIME`(UTC)。

### 3.1 ER 概览
```
(tg_user_id) ──N user_words N───1 words 1───N word_senses
                     │
                     └───1───N review_logs

readings(阅读留档,按 tg_user_id 归属)
```
> **无 `users` 表**:Telegram 每条消息自带 `tg_user_id`,直接作为 owner 列写入业务表。无认证、无注册、无配额。

### 3.2 表:`words` — 单词词条(全局去重)
```sql
CREATE TABLE words (
  id         BIGINT       NOT NULL AUTO_INCREMENT,
  text       VARCHAR(128) NOT NULL COMMENT '单词/短语原文(小写归一)',
  phonetic   VARCHAR(64)           DEFAULT NULL COMMENT '音标',
  created_at DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (id),
  UNIQUE KEY uk_text (text)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

### 3.3 表:`word_senses` — 释义/例句缓存(DictionaryAPI.dev 结果落库)
一个单词可有多个词性(meaning),每个词性存一行。
```sql
CREATE TABLE word_senses (
  id          BIGINT       NOT NULL AUTO_INCREMENT,
  word_id     BIGINT       NOT NULL,
  pos         VARCHAR(32)           DEFAULT NULL COMMENT '词性 partOfSpeech',
  meaning_en  VARCHAR(1024) NOT NULL COMMENT '英文释义 definition',
  meaning_cn  VARCHAR(512)          DEFAULT NULL COMMENT '中文释义(纯英英,保留列位不使用)',
  examples    JSON                  DEFAULT NULL COMMENT '例句数组(definition.example)',
  synonyms    JSON                  DEFAULT NULL COMMENT '同义词',
  antonyms    JSON                  DEFAULT NULL COMMENT '反义词',
  source      VARCHAR(32)  NOT NULL DEFAULT 'dictionaryapi' COMMENT 'dictionaryapi/manual',
  created_at  DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (id),
  KEY idx_word (word_id),
  CONSTRAINT fk_sense_word FOREIGN KEY (word_id) REFERENCES words(id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```
> `words.phonetic` 也由 DictionaryAPI 的 `phonetic`/`phonetics[].text` 回填;发音音频链接可加一列 `audio_url` 或塞进 `word_senses` 备用。

### 3.4 表:`user_words` — 用户单词本 + SM-2 记忆状态(核心表)
```sql
CREATE TABLE user_words (
  id            BIGINT     NOT NULL AUTO_INCREMENT,
  tg_user_id    BIGINT     NOT NULL COMMENT 'Telegram 用户 ID(owner,无 users 表)',
  word_id       BIGINT     NOT NULL,
  -- SM-2 记忆状态 --
  ease_factor   DECIMAL(4,2) NOT NULL DEFAULT 2.50 COMMENT 'SM-2 EF,最小 1.30',
  interval_days INT        NOT NULL DEFAULT 0   COMMENT '当前间隔(天)',
  repetitions   INT        NOT NULL DEFAULT 0   COMMENT '连续答对次数 n',
  due_at        DATETIME   NOT NULL COMMENT '下次到期复习时间',
  last_review_at DATETIME           DEFAULT NULL,
  status        TINYINT    NOT NULL DEFAULT 0 COMMENT '0=新词 1=学习中 2=已掌握 3=搁置',
  created_at    DATETIME   NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at    DATETIME   NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (id),
  UNIQUE KEY uk_user_word (tg_user_id, word_id),
  KEY idx_due (tg_user_id, due_at),       -- 查「今日到期」用
  CONSTRAINT fk_uw_word FOREIGN KEY (word_id) REFERENCES words(id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```
> `tg_user_id` 不建外键(无 users 表),它本身就是可信来源。

### 3.5 表:`review_logs` — 复习流水(可追溯记忆曲线)
```sql
CREATE TABLE review_logs (
  id            BIGINT   NOT NULL AUTO_INCREMENT,
  user_word_id  BIGINT   NOT NULL,
  quality       TINYINT  NOT NULL COMMENT '本次自评 0~5',
  prev_interval INT      NOT NULL,
  next_interval INT      NOT NULL,
  prev_ef       DECIMAL(4,2) NOT NULL,
  next_ef       DECIMAL(4,2) NOT NULL,
  reviewed_at   DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (id),
  KEY idx_uw (user_word_id, reviewed_at),
  CONSTRAINT fk_log_uw FOREIGN KEY (user_word_id) REFERENCES user_words(id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

### 3.6 表:`readings` — LLM 生成的阅读短文留档
```sql
CREATE TABLE readings (
  id          BIGINT       NOT NULL AUTO_INCREMENT,
  tg_user_id  BIGINT       NOT NULL COMMENT '为谁生成',
  content     TEXT         NOT NULL COMMENT 'Gemini 生成的短文',
  target_words JSON                 DEFAULT NULL COMMENT '文中嵌入的目标词列表',
  model       VARCHAR(64)           DEFAULT NULL COMMENT '生成模型标识',
  vec_id      VARCHAR(64)           DEFAULT NULL COMMENT '后续接 chroma 时的向量 id,MVP 留空',
  created_at  DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (id),
  KEY idx_user (tg_user_id, created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

> 可选表:`messages`(会话日志),先不建,MVP 后视需要补。

### 3.7 SM-2 算法说明(落到字段)
每次复习收到自评 `q ∈ [0,5]`:
```
if q < 3:                      # 答错
    repetitions = 0
    interval    = 1            # 明天重来
else:                          # 答对
    if repetitions == 0: interval = 1
    elif repetitions == 1: interval = 6
    else: interval = round(interval * ease_factor)
    repetitions += 1

ease_factor = ease_factor + (0.1 - (5-q)*(0.08 + (5-q)*0.02))
ease_factor = max(1.30, ease_factor)
due_at = now + interval days
```
> 该计算在 **Go 数据服务** 内实现(见 4.5),保证记忆状态的一致性与事务性。

## 4. Golang 微服务设计

### 4.1 职责
数据 + 调度 + **词典抓取**服务,不含任何 LLM/对话逻辑。对 Python Agent 暴露 gRPC。
词典外部调用(DictionaryAPI.dev)与缓存都收敛在这一层,Python 侧查询只需一次 gRPC。

### 4.2 分层与目录
```
service-data/
├── cmd/server/main.go          # 启动 gRPC server
├── internal/
│   ├── handler/                # gRPC handler,参数校验
│   ├── service/                # 业务逻辑:单词本、SM-2 调度
│   │   ├── word.go
│   │   └── review.go           # SM-2 计算
│   ├── dict/                   # DictionaryAPI.dev 客户端 + 响应解析
│   │   └── dictionaryapi.go
│   ├── repo/                   # SQL 访问(sqlx)
│   │   ├── user_repo.go
│   │   ├── word_repo.go
│   │   └── review_repo.go
│   └── model/                  # 结构体 / DTO
├── migrations/schema.sql       # 建表 SQL
└── go.mod
```

### 4.3 REST 端点(HTTP/JSON)
> 无用户端点:`tg_user_id` 随各请求传入,不做用户系统。所有请求/响应均 JSON。

| Method | Path | 说明 | 请求体 |
|---|---|---|---|
| `GET`  | `/words/{text}` | 查词:命中缓存直接返回;未命中 Go 内部调 DictionaryAPI.dev 解析入库后返回。404=词典无该词 | — |
| `POST` | `/notebook` | 加入某人单词本(SM-2 初始,`due_at=今天`) | `{tg_user_id, word_id}` |
| `GET`  | `/reviews/due?tg_user_id=&limit=` | 今日到期复习卡 | — |
| `POST` | `/reviews` | 提交自评 → SM-2 更新(事务) | `{user_word_id, quality}` |
| `GET`  | `/words/recent?tg_user_id=&n=` | 近期学习词,供 LLM 选材 | — |
| `POST` | `/readings` | LLM 生成短文留档 | `{tg_user_id, content, target_words}` |
| `GET`  | `/stats?tg_user_id=` | 学习统计 | — |
| `GET`  | `/healthz` | 健康检查 | — |

响应示例(`GET /words/resilient`):
```json
{
  "id": 12, "text": "resilient", "phonetic": "/rɪˈzɪljənt/",
  "senses": [
    {"pos": "adjective",
     "meaning_en": "able to recoil or spring back into shape after bending",
     "examples": ["a shoe with a durable, resilient sole"],
     "synonyms": ["tough", "hardy"], "antonyms": ["fragile"]}
  ]
}
```

### 4.4 词典抓取(service/word.go + dict/)
`LookupWord(text)`:
1. `word_repo` 按归一化文本查 `words`;有则连表取 `word_senses` 缓存直接返回。
2. 未命中 → `dict.Fetch(text)` 调 `GET https://api.dictionaryapi.dev/api/v2/entries/en/{word}`。
   - 404(词典无该词)→ 返回 `NotFound`,Agent 提示用户「未收录」。
   - 200 → 解析 `phonetic` / `meanings[].partOfSpeech` / `definitions[].definition`/`.example` / `synonyms`/`antonyms`。
3. 事务内写 `words`(含 phonetic)+ 多行 `word_senses`,返回完整词条。
> 建议对外部调用加超时(如 5s)与失败降级(返回 `503`,Bot 提示稍后重试)。

### 4.5 SM-2 调度(service/review.go 要点)
- `SubmitReview` 在**单事务**内:读 `user_words` 当前状态 → 按 3.7 公式算新 `ef/interval/due_at` → 更新 `user_words` → 写 `review_logs`。
- `GetDueWords`:`SELECT ... WHERE tg_user_id=? AND due_at<=NOW() ORDER BY due_at LIMIT ?`(无配额,`limit` 仅防一次推太多)。
- EF 下限钳制 1.30;`quality<3` 重置 `repetitions`。

### 4.6 配置与运维
- 配置走环境变量:`MYSQL_DSN`、`HTTP_ADDR`(默认 `:8080`)、`DICT_API_BASE`(默认 `https://api.dictionaryapi.dev/api/v2/entries/en`)。
- 健康检查:`GET /healthz`;建表:`migrations/schema.sql`。

## 5. AI Agent 程序设计

### 5.1 形态
基于 **langgraph** 的单 Agent + 工具集;Telegram 消息进入后先做**意图路由**(查询 / 复习 / 推荐 / 闲聊),再调对应工具。Agent 不直接读写 MySQL,全部经 HTTP 调 Go。

### 5.2 工具(Tools)
| 工具 | 说明 | 依赖 |
|---|---|---|
| `lookup_word(text)` | `GET /words/{text}`(词典抓取+缓存都在 Go 内)+ `POST /notebook`。**不调 LLM** | HTTP |
| `start_review(user)` | `GET /reviews/due`,逐张返回卡片 | HTTP |
| `grade_review(uw_id, quality)` | `POST /reviews`,回下次复习时间 | HTTP |
| `recommend_reading(user)` | `GET /words/recent` → Gemini 生成含目标词短文 → `POST /readings` 留档 | HTTP + Gemini |

### 5.3 记忆(两层)
- **长期记忆**:用户单词本 + SM-2 状态,存于 MySQL(经 Go)。这是「记忆系统」的持久层。
- **短期记忆**:langgraph 的会话 checkpoint(按 `tg_user_id` 维护 thread),保证多轮对话上下文(如「它的例句再来一个」)。

### 5.4 查询链路(时序)
```
用户发 "resilient"
  → Bot handler → Agent 意图=查询
  → lookup_word:
       GET /words/resilient               # Go 内部:
         命中缓存? 是 → 直接返回词条
                   否 → 调 DictionaryAPI.dev → 解析入库 → 返回词条
       POST /notebook {tg_user_id, word_id}   # SM-2 初始: due_at=今天
  → 格式化卡片(音标/英文释义/例句/同义词)回复 TG
```
> LLM(Gemini)在查询链路**不参与**;仅用于「推荐阅读」链路(5.5)。

### 5.5 推荐阅读链路
```
/read
  → GET /words/recent?tg_user_id=&n=10
  → Gemini 生成一段自然、含这些目标词的定制短文
  → POST /readings 留档
  → 回复 TG(短文 + 文中出现的目标词列表)
```

### 5.6 代码结构(Python 侧)
- `bot/`:python-telegram-bot 入口、命令与消息 handlers。
- `agent/`:langgraph 意图路由、tools、prompts。
- `clients/data_api.py`:Go 服务 HTTP 客户端(httpx)。
- 复用现有 `.env` 的 `MODEL`、`MAX_RETRIES`、`LOG_LEVEL` 与重试逻辑;新增 `TG_BOT_TOKEN`、`DATA_API_BASE`。

## 6. 端到端示例(复习一次)
1. 用户 `/review` → Agent `start_review` → Go `GetDueWords` 返回 3 张卡。
2. Bot 推送卡片 1(遮释义),用户点「模糊」= quality 3。
3. Agent `grade_review(uw_id, 3)` → Go 在事务内跑 SM-2:`interval 1→6`,`due_at=+6天`,写 `review_logs`。
4. Bot 回「下次复习:6 天后」,推下一张。

## 7. 里程碑
- **M1 数据层**:MySQL 建表 + Go 数据服务(单词/单词本/SM-2 + 词典抓取)+ REST 打通。
